# Intelligence Core v2 Design

## Goal

Turn 情报官 from a search-and-report task center into an intelligence-analysis system with enforced evidence discipline, temporal fact management, hypothesis testing, and auditable reports.

## Problem

Current investigations can contain useful entities, evidence, relationships, graph nodes, memory findings, and reports, but the discipline is not enforced end to end:

- Confirmed contacts can exist in entities but be omitted from report body.
- Report statements are not required to cite evidence anchors.
- Confirmed facts, weak claims, hypotheses, and risk signals can blur together.
- Old facts and new facts overwrite each other instead of forming a time-aware record.
- ACH and Admiralty Code utilities exist, but they are not mandatory gates before analysis output.
- Agent jobs can remain as queue labels instead of becoming closed-loop directed collection.

## External References

Use these as architecture references, not hard runtime dependencies:

- OpenOSINT (`https://github.com/OpenOSINT/OpenOSINT`): AI-powered OSINT agent with interactive REPL, MCP server, CLI, managed subprocess tools, and structured reports. Borrow its tool-router boundaries and MCP exposure style, but keep our own evidence and report contract.
- OWASP Social OSINT Agent (`https://github.com/OWASP/www-project-social-osint-agent`): OWASP project web repository describing a SocialOSINTAgent that gathers public social data through official APIs, injects the current UTC timestamp into analysis prompts, supports cached/offline analysis, and outputs structured reports. Borrow time-awareness and ethical collection constraints.
- madeinoz-osint-skill (`https://github.com/madeinoz67/madeinoz-osint-skill`): Claude OSINT skill/pack with workflow-driven investigations, knowledge graph persistence, and an explicit five-step cycle: Planning, Collection, Processing, Analysis, Storage. Borrow the cycle discipline and dual storage idea.
- Graphiti by Zep (`https://github.com/getzep/graphiti`): temporal context graph engine for AI agents. Borrow temporal fact windows, episode/provenance concepts, incremental graph updates, hybrid retrieval, and MCP-server memory shape.
- HypoGeniC / HypoRefine (`https://github.com/ChicagoHAI/hypothesis-generation`): official repository for data-driven and literature-plus-data hypothesis generation. Borrow generated hypothesis banks and refinement loops as upstream input to ACH.
- LLM-SCI-GEN (`https://github.com/Paureel/LLM-SCI-GEN`): curated papers on scientific hypothesis generation using LLMs. Use as literature reference, not a direct runtime dependency.

Repository notes:

- OpenOSINT is useful for MCP/tool-loop architecture, but many tools target security reconnaissance. Any integration must be restricted to authorized, legal, public-source investigations.
- OWASP Social OSINT Agent's GitHub repo is a website repository, not necessarily the full implementation repository; use its documented design principles and verify implementation code separately before importing anything.
- madeinoz-osint-skill is small but directly relevant as workflow inspiration. Treat it as a pattern source, not a dependency.
- Graphiti is the strongest candidate for future optional graph backend exploration, but Core v2 should first implement temporal facts in SQLite to keep deployment light on N100.

## Core Concepts

### Fact Pool

The Fact Pool stores assertions the system is willing to treat as facts or likely facts. Every fact must include:

- `statement`: atomic assertion.
- `subject`, `predicate`, `object`: normalized triple.
- `status`: `CONFIRMED`, `LIKELY`, `CONTRADICTED`, `RETIRED`, `NEEDS_REVIEW`.
- `confidence`: numeric probability 0-1.
- `admiralty_code`: source reliability and information credibility.
- `evidence_ids`: one or more evidence anchors.
- `observed_at`: when the system observed the information.
- `valid_from` and optional `valid_to`: temporal validity window.
- `supersedes_fact_id`: optional previous fact.

No report can call a claim "confirmed" unless it exists in the Fact Pool with evidence.

### Hypothesis Pool

The Hypothesis Pool stores competing explanations or judgments. Every analysis task that makes a judgment must create at least three mutually exclusive hypotheses unless the task is purely clerical.

Each hypothesis includes:

- `statement`.
- `mutually_exclusive_group`.
- `status`: `MOST_LIKELY`, `PLAUSIBLE`, `DISFAVORED`, `REJECTED`, `UNVERIFIED`.
- supporting and contradictory evidence IDs.
- ACH scores generated by deterministic code.

Unverified leads must live here, not in the Fact Pool.

### Evidence Ledger

Evidence Ledger is the source-of-truth for provenance. Each evidence item stores:

- source URL or source descriptor.
- source type.
- collected/observed timestamp.
- snippet or structured payload.
- source reliability class.
- information credibility class.
- hash of snippet/payload for dedupe.
- entity/fact/hypothesis links.

Evidence is never deleted during normal operation. Newer evidence can supersede earlier interpretation.

### Temporal Graph

The graph becomes a view over facts, hypotheses, evidence, sources, and relationships. It should support:

- current-state view.
- historical timeline view.
- source-chain view.
- contradiction view.
- unknown/gap view.

Old facts are marked retired or superseded instead of overwritten.

### Report Audit Trail

Every material report section must be generated from structured records:

- BLUF must cite key facts and hypotheses.
- Contact section must include all confirmed and high-confidence candidate contacts.
- Confirmed facts table must cite evidence IDs.
- ACH section must include competing hypotheses and contradiction scores.
- Risk/I&W section must cite indicators and evidence.
- Directed collection section must list unresolved gaps and exact next actions.

Any report sentence containing a named entity, contact, risk signal, or judgment should map to at least one evidence anchor.

## Workflow

The enforced workflow is:

1. Planning: define PIR, scope, constraints, and initial hypotheses.
2. Collection: gather sources through tools or manual import.
3. Processing: normalize entities, contacts, evidence, source reliability, and timestamps.
4. Verification: assign Admiralty Code, dedupe, separate facts from hypotheses.
5. ACH Analysis: score competing hypotheses using evidence support/contradiction.
6. Memory Update: write Fact Pool, Hypothesis Pool, Evidence Ledger, temporal graph, and collection gaps.
7. Reporting: generate BLUF and fixed sections from structured records.
8. Directed Collection: create follow-up jobs from unresolved gaps.

## Data Model Additions

Add SQLite-backed tables:

- `facts`
- `hypotheses`
- `hypothesis_evidence`
- `evidence_links`
- `source_records`
- `report_anchors`
- `collection_gaps`

Existing `entities`, `evidence`, `relationships`, and `jobs` remain for compatibility. Core v2 derives facts and hypotheses from them during migration and future writes.

## Agent Protocol Additions

Add endpoints or protocol commands for:

- writing facts.
- writing hypotheses.
- linking evidence to facts/hypotheses.
- requesting ACH scoring.
- completing a report with audit anchors.
- opening/closing collection gaps.

Agent writes should fail validation if:

- a fact lacks evidence.
- a report has required sections missing.
- a confirmed contact is present in Fact Pool but omitted from the contact report section.
- an analysis judgment lacks competing hypotheses.

## UI Additions

Keep the dense operations-console style from `DESIGN.md`.

Add panels:

- Fact Pool: confirmed, likely, contradicted, retired.
- Hypothesis Pool / ACH Matrix.
- Evidence Ledger with source reliability and timestamps.
- Report Anchors / audit trail.
- Collection Gaps and directed follow-up jobs.

The main graph remains curated, but the right-side or lower panels must expose all structured findings and report-critical details.

## Report Contract

Enterprise reports must include these sections:

1. BLUF.
2. Key Contacts and Contact Channels.
3. Confirmed Facts.
4. Business Scope and Operating Footprint.
5. News / Events.
6. Risk Signals.
7. ACH / Alternative Hypotheses.
8. Evidence Table.
9. Directed Collection.

For sparse buyer leads, use the same structure with buyer-intent specific labels.

## Testing Requirements

Tests must prove:

- facts cannot be confirmed without evidence.
- hypotheses can be scored with ACH and contradictory evidence.
- report generation includes every confirmed contact.
- report anchors point to valid evidence IDs.
- temporal supersession preserves old facts.
- collection gaps are created for missing contacts, news, risk signals, or hard-asset proof.
- existing investigations remain readable after migration.

## Non-Goals

- Do not replace the current simple SQLite store with a graph database in this phase.
- Do not depend on external paid APIs for core tests.
- Do not scrape private data or bypass access controls.
- Do not make the graph visually crowded; preserve curated slots plus detailed panels.

## Acceptance Criteria

- A new investigation can complete the full Core v2 workflow locally.
- SRR-style company investigations generate contact-inclusive reports from structured facts.
- A report cannot mark a fact confirmed unless it has evidence and an Admiralty Code.
- The UI shows Fact Pool, Hypothesis Pool, Evidence Ledger, report anchors, and collection gaps.
- Existing tests pass, and new Core v2 tests cover the contract above.
