# 通用调查能力加固设计

**日期：** 2026-07-13  
**状态：** 已确认，待实施计划  
**范围：** 官网 fake-IP 审批、业务范围提取、联系方式分级、验收基准和工具能力影响可观测性。

## 背景与结论边界

当前平台已经具备稳定的作业、证据账本、质量闸门和已知域名调查能力，但不能据此声称任意公司、行业或弱线索均可自动完成调查。本次只改善可证实的覆盖面、失败可解释性和验收方法；不承诺消除真实网站访问限制，也不把未验证信息升级为事实。

当前实现的主要约束如下：

- `safe_http` 仅在精确主机名和限定 fake-IP 网段同时匹配时允许请求。这是正确的默认安全边界，但新目标被透明网络映射时只会得到通用抓取失败。
- 官网业务范围仅匹配固定英文词组，无法可靠覆盖其他行业、中文页面或依赖 Schema.org 的站点。
- 官网页面中的电话和邮箱没有区分传真、客服、总机、通用公开通道或与具名职位关联的联系人。
- `regression_smoke.py` 使用合成实体，验证报告契约而不是网页访问、任务状态或事实准确性。
- 工具健康状态可见，但缺少“某个不可用工具会降低哪些调查能力”的汇总。

## 目标与非目标

### 目标

1. 对 fake-IP 阻断生成稳定、非敏感、可审计的人工审批信号，同时保持默认拒绝。
2. 用确定性、可追溯的结构化页面信号提取跨行业业务范围，并扩展中文及其他非英语结构化页面的覆盖。
3. 将公开联系方式分级；只有姓名、职位和同一上下文联系方式同时出现时才建立角色关联。
4. 明确区分离线契约回归和真实端到端验收，并为三种种子类型输出可比较指标。
5. 在工具健康输出和任务报告中说明不可用工具导致的能力缺口。

### 非目标

- 不放开所有 fake-IP 主机名，不允许通配符、IP 字面量或自动批准。
- 不在本轮拨打电话、投递邮件、登录第三方账号或把静态页面证据描述为可触达性验证。
- 不接入未授权的外部 LLM、搜索、代理或数据经纪服务。
- 不把真实验收运行器的结果当成对所有客户自动调查能力的证明。
- 不在本轮安装或配置 N100 上缺失的第三方工具；代码只说明其覆盖影响。

## 方案选择

考虑过三种方向：

1. 只补充失败提示和运维说明。它不会改善业务范围、联系人准确性或真实验收，不能满足整改目标。
2. 保持安全限制，新增审批、结构化提取、联系人分级和显式验收基准。该方案可在本仓库内测试，且不依赖未授权外部服务，采用此方案。
3. 放开 fake-IP 主机名或接入外部验证服务。前者把 SSRF 边界交给透明网络配置，后者需要额外授权、成本、隐私与误判治理，均不作为默认路径。

## 架构设计

### 1. fake-IP 审批清单和阻断结果

新增可选的审批文件环境变量 `OSINT_SAFE_HTTP_FAKE_IP_APPROVALS_FILE`。文件是受运维变更控制的 JSON，包含一个受限 IPv4 fake-IP 网段列表以及逐主机审批记录：

```json
{
  "version": 1,
  "networks": ["198.18.0.0/15"],
  "approvals": [
    {
      "hostname": "www.example.com",
      "approved_by": "ops-reviewer",
      "approved_at": "2026-07-13T00:00:00Z",
      "expires_at": "2026-10-11T00:00:00Z",
      "reason": "Public official site verified by operator"
    }
  ]
}
```

规则如下：

- 主机名先经 IDNA 规范化，必须精确匹配；不支持通配符或子域推断。
- 网段必须是 `198.18.0.0/15` 的子网；IP 字面量始终拒绝。
- 审批记录必须有审核人、原因、创建时间和未过期的到期时间；无效、重复或过期配置一律失败关闭。
- 配置文件模式和旧的 `OSINT_SAFE_HTTP_FAKE_IP_CIDRS` / `OSINT_SAFE_HTTP_FAKE_IP_HOSTS` 模式不可同时启用，避免歧义。旧模式保留兼容性并在部署文档中标记为迁移路径。
- 当解析到 fake-IP 且主机未经审批时，`safe_http` 抛出专用的审批异常。`official_site_extractor` 把它转换为稳定的 `fake_ip_review_required` 失败码和仅含已请求主机名的诊断，不暴露解析器、内部路由或原始网络细节。
- 重定向中的每个主机都独立校验。任何未审批跳转仍然失败。

这不会自动提高任意新官网的抓取成功率；它会让操作员能区分“网络审批缺失”与普通 HTTP/DNS 失败，并在审核后以可追溯方式放行目标。

### 2. 业务范围的结构化候选提取

扩展 HTML 解析器以保留以下来源及其原文上下文：

- JSON-LD 的 `description`、`knowsAbout`、`makesOffer`、`hasOfferCatalog`、`itemListElement`、`Product.name`、`Product.category` 和 `Service.name`；
- `meta[name=description]`、`meta[property=og:description]`；
- `h1` 到 `h3` 的文本及其相邻正文段；
- 现有可见正文中的有限上下文段。

候选对象包含值、来源类别、原文片段和确定性分数。结构化产品/服务字段优先；描述和标题必须包含具体产品、服务、行业或非泛化名称。中文业务提示词与英文业务提示词用于从正文定位候选段，但 JSON-LD、元描述和标题路径不依赖页面语言。导航、Cookie 文本、联系方式、过短标题和只含“solutions”、“quality”一类泛化词的候选会被丢弃。

输出仍使用 `business_scope` 实体，但证据类型按来源区分，例如 `official_site_business_scope_json_ld`、`official_site_business_scope_meta`、`official_site_business_scope_heading` 和 `official_site_business_scope_text`。证据片段保留具体的官网文字，避免把内部推断当成来源事实。现有固定模式保留为低风险回退，不再是唯一途径。

质量闸门不会因为任意长描述自动通过：只有包含具体业务内容且有来源证据的范围候选才能支撑业务范围字段。

### 3. 联系方式分级

抽取阶段改为先产生带上下文的联系人候选，再输出实体、证据和关系。

| 类型 | 输出 | 可满足的语义 |
| --- | --- | --- |
| `role_linked` | 邮箱/电话实体、官网证据、`person_has_role_linked_contact` 关系 | 公开的具名职位关联联系方式；可触达性仍未验证 |
| `public_general` | 邮箱/电话实体和官网公开通道证据 | 来源支撑的公开联系渠道 |
| `customer_service` | 邮箱/电话实体和客服标签证据 | 客服通道，不是决策人联系方式 |
| `fax` | `fax` 实体和传真证据 | 传真，仅作参考，不计入电话渠道 |

分类依据仅限页面中联系方式周边的静态标签、JSON-LD 字段和已经识别的具名人员上下文。名字、职位和联系方式必须在同一受限上下文中才生成角色关联；泛用电话、传真和客服不能反向提升决策人状态。输出中不出现“已拨通”“已投递”“已验证可达”等措辞。

### 4. 验收分层与真实样本运行器

现有 `regression_smoke.py` 继续作为离线契约回归，但输出显式标记 `suite_kind: synthetic_contract`，并验证其不做网络访问。新增多行业、多语言官网 HTML 夹具覆盖业务范围和联系人分类。

新增一个默认干跑的真实验收运行器，读取由操作员维护的案例清单。它只在显式 `--execute`、目标 API 地址和案例清单同时给出时提交或轮询调查；默认模式只校验清单和预期结构。案例清单按以下种子类型分层：

- `domain`：已知官网域名；
- `company`：公司名，需要官网发现与身份边界；
- `sparse_lead`：弱线索，需要身份、主体和业务范围收敛。

每个案例记录公开、获授权的种子，预期最低证据类型，是否允许人工审批，以及不应被视为成功的限制。运行器输出 JSON 和 Markdown 汇总，至少包含：终态分布、完成率、误冲突率、人工介入率、证据地板满足率、失败原因和受影响能力。真实运行器不把网络不可达、审批待办或工具缺失重写为成功。

没有足够的真实案例时，输出必须明确为“基准尚未建立”，不能从合成回归推导泛化完成率。

### 5. 工具能力影响

在 `tool_health` 的每个工具状态及汇总中增加确定性的 `coverage_areas` 与 `affected_capabilities`。例如：

- `theharvester`、`phoneinfoga`：联系方式发现或验证辅助；
- `maigret`、`socialscan`、`spiderfoot`：社媒及公开身份覆盖；
- `amass`、`spiderfoot`：资产与域名覆盖；
- 海关供应链工具：供应链证据覆盖。

健康报告只陈述已知能力影响，不推断某个特定目标一定缺少该信息。作业的收尾报告将附带相关不可用工具和影响类别，使人工审核者看到“为什么覆盖面下降”。第三方工具的安装、凭据和生产配置仍是单独的运维审批事项。

## 数据流与错误处理

1. 官网 URL 进入安全 URL 校验。
2. 公网地址正常抓取；fake-IP 地址只有经审批的精确主机可抓取。
3. 未审批 fake-IP 生成可审计的审批待办；其他安全错误保持通用、安全的错误边界。
4. 成功 HTML 提取结构化业务范围和分级联系方式，并为每个输出附上官网原文证据。
5. 质量与完成策略只消费来源支撑的事实，不把联系人等级或工具缺口隐式转换为成功。
6. 离线回归验证解析与报告契约；真实验收运行器仅在操作员显式执行时测试完整链路。

所有新解析器对畸形 JSON-LD、非法配置、过期审批、无效时间、超长文本和没有匹配项采用空结果或安全失败，不抛出可导致作业崩溃的未处理异常。

## 验收与验证

实施后必须满足以下可自动验证条件：

1. 无审批、错误主机、过期审批、通配符、直接 fake-IP 字面量和未审批重定向均被拒绝。
2. 有效审批只放行批准主机和受限网段；诊断只返回稳定失败码及规范化主机名。
3. 英文工业站、中文服务站及非英语 JSON-LD 夹具均能提取有来源的具体业务范围；导航和泛化词不会形成业务范围实体。
4. 传真不生成 `phone`；客服和通用通道不产生人物联系方式关系；具名职位同上下文联系方式产生角色关联且不声称可达。
5. 合成回归输出显式标记，真实验收运行器在未加 `--execute` 时绝不访问网络。
6. 工具健康报告给出不可用工具的能力影响，并保持现有状态与可执行工具选择逻辑。
7. 后端全量测试、离线回归、前端测试和生产构建均通过；真实验收仅在获授权的环境与案例清单上单独执行。

## 部署与回滚

本次不变更 N100。部署前由运维将现有精确主机白名单迁移为审批文件，复核每个主机的所有权、原因、审核人和到期时间，并重启服务使配置生效。发生异常时，移除审批文件环境变量即可回退到既有的精确主机环境变量模式；绝不以放开所有主机名作为回滚手段。

真实验收案例及其结果应保留在受访问控制的运维位置；仓库只保存无敏感目标的清单模式和合成夹具。
