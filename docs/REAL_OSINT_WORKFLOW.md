# 真实公开源信息整理固定工作流

本工作流适用于 Alibaba/CRM 弱线索买家、企业背调、决策人补全和双核心图谱整理。目标是尽最大能力补齐企业名称、官网、邮箱、电话、地址、注册信息、业务范围、贸易信号和决策人候选，同时保留证据链和不确定性。

## 1. 输入锚点

先把截图或 CRM 可见字段作为事实锚点写入任务：

- 平台、显示名、会员 ID、国家/地区、买家等级。
- 公司字段、姓名字段、邮箱/电话隐私状态。
- 注册时间、建档时间、询盘来源、业务员。
- 采购品类、RFQ、备注和授权状态。

字段标签优先于字面直觉。公司字段即使看起来像人名，也先作为公司/商号候选处理。

## 2. 约束式检索

从强约束到弱约束逐层检索：

- `"完整姓名" + 国家/城市`
- `"完整姓名" + company/import/purchasing`
- `"公司字段" + 国家/城市`
- `"姓名变体" + 行业/产品语境`
- `"候选企业名" + contact/email/phone/address/NIT/RUES`
- `"候选企业名" + import/export/supplier/China`

每次检索必须记录使用了哪些约束，不能只用泛姓名结果写主结论。

## 3. 候选企业发现

优先寻找：

- 官方网站。
- 工商/RUES/NIT/注册目录。
- D&B、Empresite/eInforma、La Republica、行业目录。
- 企业联系页、产品页、关于我们。
- 海关/进口记录公开页。

公开记录真实性和与 Alibaba 买家的同一性必须分开标注。

## 4. 联系方式与资产补齐

必须尽力补齐并写入图谱：

- 企业名称和名称变体。
- 官网和关键页面。
- 邮箱、电话、WhatsApp、地址。
- 注册号/NIT/RUES、法人形态、状态。
- 主营业务、产品线、制造/经营足迹。
- 进口/供应商/采购相关公开信号。
- 决策人姓名、职位、公开主页候选。

## 5. 工具使用顺序

能用的工具都要用，但必须按线索类型调度：

- 没有域名时，先做公开搜索和目录源补全。
- 有域名后，再跑 theHarvester、Amass、SpiderFoot、Recon-ng 路线。
- 有用户名后，再跑 Sherlock / Maigret / socialscan。
- 有邮箱后，再跑 socialscan / SpiderFoot / GHunt（如果配置可用）。
- 有电话后，再跑 PhoneInfoga。

缺命令或缺配置必须写入队列/事件，不得静默跳过。

## 6. 写回标准

所有发现必须写成：

- `Entity`
- `Evidence`
- `Relationship`
- `Evidence Ledger`
- `Fact`
- `ACH Hypothesis`
- `BLUF Report`

图谱主槽位优先显示交叉验证后的企业、官网、联系方式、主营业务和决策人候选。弱来源、同名噪声和工具原始命中进入候选/待复核。

## 7. 结论语言

使用这些标签：

- `CONFIRMED`：官网、注册源或强交叉验证确认。
- `LIKELY`：多源支持但仍缺直接闭环。
- `CANDIDATE`：工具命中或单源候选。
- `NEEDS_REVIEW`：需要人工或买家回复确认。
- `CONFLICT`：公开来源之间存在矛盾。

不得把同名公开记录直接写成账号本人。报告必须说明“公开记录真实性”和“与该买家的同一性”。

## 8. 验收门槛

每次信息整理至少检查：

- 企业名称是否补齐。
- 官网是否补齐。
- 邮箱/电话/地址是否补齐。
- 注册信息是否补齐。
- 主营业务/产品是否补齐。
- 决策人候选是否有证据。
- 图谱是否同步显示关键字段。
- 证据账本和事实池是否承载关键结论。
- 报告是否包含 BLUF、ACH、缺口和下一步动作。
