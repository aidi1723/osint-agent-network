# Security Audit Remediation Ledger

## Scope and usage

This ledger tracks the eight findings accepted in the 2026-07-10 security-hardening design. Update the relevant row immediately after its targeted regression test turns green, and close a finding only after the complete verification suite has passed. A `planned` status records intent only and does not claim that the finding is fixed.

| ID | Severity | Original reproduction / evidence | Target regression test | Implementation status | Verification evidence | Residual risk |
| --- | --- | --- | --- | --- | --- | --- |
| SEC-01 | High | Building the frontend with `VITE_ADMIN_API_TOKEN` embeds the long-lived management token in browser assets. | Frontend auth/build regression: assert browser code constructs no bearer header from environment tokens and scan production assets for credential values. | planned | Planned: targeted frontend auth test plus production build asset scan; not run yet. | Browser credential exposure remains until cookie-based admin sessions replace environment-token access and the asset canary passes. |
| SEC-02 | High | External agents share one bearer token, while request payloads can supply a forged `agent_id` or role without identity, role, and claim-ownership binding. | Backend agent authorization matrix: cover token hashing and rotation, forged identity, wrong role, unregistered agent, claim ownership, and legacy-mode production rejection. | planned | Planned: targeted backend agent authentication/authorization tests; not run yet. | Cross-agent impersonation and unauthorized task mutation remain possible until per-agent credentials and all route/ownership checks are verified. |
| SEC-03 | High | Current URL validation permits private-network SSRF variants and does not safely revalidate redirect targets. | Safe-fetch regression suite: cover IPv4/IPv6 private ranges, alternate loopback forms, mixed DNS answers, redirect hops, response limits, and an injected public-safe connection. | planned | Planned: targeted safe-fetch and target-normalization tests with injected resolver/connector fakes; not run yet. | Private-network access or redirect-based policy bypass remains possible until DNS pinning, redirect validation, and limits are verified. |
